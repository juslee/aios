## §9 Subsystem Thermal Coordination

Each AIOS subsystem coordinates with the thermal management system via the Policy Engine. Subsystems do not independently throttle — they respond to thermal directives from the Policy Engine. This centralised model ensures that thermal responses across subsystems are coherent and prioritised correctly: the kernel decides which subsystems absorb heat budget reductions and in what order, rather than each subsystem making local decisions that may conflict or over-react.

The Policy Engine issues thermal directives as structured events on each subsystem's thermal channel. Subsystems subscribe to these events during initialisation. The directive specifies the new thermal state, the recommended cooling action, and an urgency level that determines how quickly the subsystem must comply.

### §9.1 GPU Thermal Coordination

The GPU is typically the highest heat source after the CPU and the first subsystem targeted by thermal throttling. Its power consumption scales steeply with frequency, making frequency reduction the most effective lever for heat reduction.

**Predictive Throttling:**

Reactive throttling — waiting until a threshold is crossed — produces abrupt frame rate drops that users perceive as stutter. The thermal integration layer uses gradient-based prediction to begin frequency reduction before the threshold is reached.

- Compute thermal gradient: `gradient = (temp_now - temp_5s_ago) / 5.0` (°C/s, where temperatures are in millidegrees Celsius divided by 1000)
- Predict future temperature: `predicted_10s = temp_now + gradient * 10.0`
- If `predicted_10s > throttle_threshold`: reduce GPU frequency by one step now, before the threshold is crossed
- If `temp_now > (throttle_threshold - 5_000)` (i.e., within 5°C of threshold): cap GPU at 75% of maximum frequency

The throttle threshold is derived from the zone's passive trip point (zones.md §3.2).

**UX Impact:**

| Approach | FPS Transition | User Perception |
|---|---|---|
| Reactive throttling | 60 → 30 fps (abrupt) | Jarring stutter |
| Predictive throttling | 60 → 45 → 30 fps (smooth) | Gradual quality reduction |

Predictive throttling trades a small amount of sustained performance for perceptual smoothness. The 10-second prediction horizon is long enough to act before the trip is crossed, but short enough to avoid premature throttling during transient workloads.

**GPU-CPU Thermal Coupling:**

Heat generated by the GPU raises the temperature of adjacent CPU zones through conductive and convective coupling (zones.md §3.4 coupling model). The thermal integration layer accounts for this when sequencing subsystem throttling.

- When the GPU zone reaches Passive: reduce GPU frequency first; monitor CPU zone temperature
- If CPU zone continues rising despite GPU reduction: begin CPU frequency reduction
- Priority order preserves interactive responsiveness (CPU scheduler, compositor, input) over GPU compute throughput

Reference: `gpu/integration.md §17.2.4` for GPU thermal prediction implementation details.

### §9.2 Audio Thermal Coordination

Audio is the most latency-sensitive subsystem in AIOS. Glitches in audio output are immediately perceptible to users in a way that GPU or storage slowdowns are not. Accordingly, audio is the last subsystem to be throttled under thermal pressure and the first to have restrictions lifted during thermal recovery.

**Thermal Priority:**

Audio maintains its full allocated CPU budget until `ThermalState::Critical` is reached. The audio real-time threads run at the highest non-kernel scheduling priority and are not subject to thermal quota scaling until the system is in crisis.

- At `Warm`: audio processing is entirely unaffected
- At `Throttled`: audio real-time deadlines are scaled (scheduling.md §6.3) but callbacks are never dropped; latency may increase slightly
- At `Critical`: system audio continues (shutdown chime, accessibility alerts) but general playback is halted to prevent feedback loops where audio DSP heat prevents thermal recovery

**Power Reduction Strategies:**

When thermal directives do require audio to reduce its power footprint, actions are applied in order of increasing user impact:

- Reduce DSP filter chain complexity: bypass non-essential effects (reverb, spatial audio enhancement) while preserving equalisation and volume normalisation
- Lower sample rate from 48 kHz to 44.1 kHz if the active audio content permits the quality trade-off
- Disable audio capture pipelines (microphone, line-in) when no agent holds an active capture capability

Reference: `audio/integration.md §9` for audio power management implementation details.

### §9.3 Storage Thermal Coordination

Storage subsystem heat generation is lower priority relative to CPU, GPU, and audio. Background I/O — compaction, garbage collection, prefetch — can be freely deferred without immediate user impact. Foreground I/O (user-initiated reads and writes) is protected from thermal throttling until `ThermalState::Critical`.

**Thermal Actions by State:**

- At `Warm`: reduce WAL compaction frequency; defer MemTable flush to next idle window (block-engine.md §4)
- At `Throttled`: pause background garbage collection entirely; reduce write throughput for background tasks by inserting inter-write delays; foreground I/O continues unthrottled
- At `Critical`: allow all in-flight writes to complete (data safety invariant); then pause all new background I/O until thermal state drops to Throttled or below

**Write Amplification Trade-off:**

Deferring LSM-tree compaction under thermal pressure increases write amplification when compaction eventually runs. The thermal integration layer schedules a catch-up compaction burst during the first sustained thermal recovery period following a Throttled or Critical episode.

On platforms with NVMe or eMMC storage that report their own thermal zones, storage device temperature is monitored independently. If the storage zone approaches its own trip point, foreground I/O is also throttled to protect the device.

Reference: `storage/spaces/block-engine.md §4` for WAL and LSM-tree compaction details.

### §9.4 Networking Thermal Coordination

Network I/O generates moderate heat primarily through DMA transfers and interrupt processing. The networking thermal integration focuses on reducing interrupt frequency and DMA burst size rather than dropping throughput.

**Thermal Actions by State:**

- At `Warm`: increase interrupt coalescing parameters (batch more packets per IRQ, accepting slightly higher latency); no impact to application-visible throughput
- At `Throttled`: reduce TCP receive window sizes to slow inbound data rate; defer prefetch operations on predictive connections; reduce number of active connection poll timers
- At `Critical`: maintain only essential connections (DNS resolver, NTP, active system update downloads); suspend all background sync and telemetry channels

**Wi-Fi Specific:**

Wi-Fi radio transmit power is a significant heat source on mobile platforms.

- At `Warm`: reduce transmit power by one step (typically 1–3 dBm reduction); adequate for most indoor distances
- At `Throttled`: switch from high-throughput (HT/VHT) mode to legacy mode, sacrificing throughput for power
- At any throttled state: suspend all background network scans; scans are rescheduled for the next thermal recovery window

Reference: `networking.md §3` for NTM component architecture and connection management.

### §9.5 Boot-Time Thermal Behaviour

The system may be warm at boot time due to a prior thermal shutdown, a hot ambient environment, or rapid reboot after heavy workload. The thermal subsystem must begin monitoring before the full kernel is initialised.

**Boot Thermal Protocol:**

1. Phase 2 (HAL init): start thermal sensor polling immediately after the driver framework initialises; establish baseline temperature readings for all zones
2. If any zone is at or above the Passive trip point at boot:
   - Emit UART warning: `"Boot thermal: zone {name} at {temp_mc}mc"` (millidegrees Celsius)
   - Delay AIRS model loading — the initial model load is a large compute burst that would worsen an already-warm thermal state
   - Proceed with minimal services only until at least one zone reading drops 5°C below the Passive trip
3. If any zone is at or above the Hot trip point at boot:
   - Enter reduced boot mode
   - Skip the GPU framebuffer test pattern (eliminates an unnecessary GPU heat burst during boot)
   - Defer secondary core bringup until primary core thermal zone drops below Passive
4. If any zone is at or above the Critical trip point at boot:
   - Halt the boot sequence immediately
   - Emit UART message: `"System too hot — please wait"`
   - Poll thermal sensors in a lightweight busy-idle loop until all zones drop below `(passive_trip - hysteresis_mc)` where `hysteresis_mc` is the zone's configured hysteresis margin (zones.md §3.3)
   - Resume boot from the beginning of Phase 2 once the thermal condition clears

This protocol ensures the kernel never attempts to initialise power-hungry subsystems (SMP, GPU, AIRS) when doing so would immediately trigger thermal shutdown.

Reference: `hal.md §17.4` for boot-time thermal monitoring integration with the HAL initialisation sequence.

## §10 External Interfaces

### §10.1 POSIX Thermal Interfaces

AIOS exposes thermal zones through a sysfs-compatible virtual filesystem to support existing Linux userspace thermal tools (`thermald`, `lm-sensors`, system monitors) without modification.

**Filesystem Layout:**

```text
/sys/class/thermal/
├── thermal_zone0/
│   ├── type              → "cpu-thermal"
│   ├── temp              → "80000"  (millidegrees Celsius as decimal string)
│   ├── trip_point_0_temp → "80000"
│   ├── trip_point_0_type → "passive"
│   ├── trip_point_1_temp → "85000"
│   ├── trip_point_1_type → "hot"
│   ├── trip_point_2_temp → "90000"
│   ├── trip_point_2_type → "critical"
│   └── policy            → "pid"
├── thermal_zone1/
│   ├── type              → "gpu-thermal"
│   ├── temp              → "72000"
│   ├── trip_point_0_temp → "80000"
│   ├── trip_point_0_type → "passive"
│   ├── trip_point_1_temp → "90000"
│   ├── trip_point_1_type → "critical"
│   └── policy            → "pid"
└── cooling_device0/
    ├── type              → "dvfs"
    ├── max_state         → "3"
    ├── cur_state         → "0"
    └── driver_name       → "cpu-cooling"
```

**Access Control:**

- Read access to `temp` and `trip_point_*_temp` files: requires `ThermalCapability::ThermalRead`
- Write access to trip points and `policy`: requires `ThermalCapability::ThermalConfigure`
- `cur_state` on cooling devices is read-only for all agents; only the Policy Engine may change cooling device state

The `temp` file is updated on each read (not cached) to ensure monitoring tools receive fresh readings. The POSIX filesystem layer translates the internal `i32` millidegrees Celsius representation to a decimal string, matching Linux sysfs convention.

Reference: `posix.md §9` for device file conventions and virtual filesystem implementation.

### §10.2 Agent Thermal Headroom API

The thermal headroom API allows agents to query their remaining thermal budget and proactively reduce workload before the kernel is forced to throttle them. This cooperative model produces smoother user experiences than purely reactive kernel-imposed throttling.

The design is inspired by Android's Dynamic Performance Framework (ADPF) thermal headroom API, adapted for AIOS's capability-gated agent model.

**Data Structures:**

```rust
/// Thermal headroom information returned to agents.
///
/// All temperature values are in millidegrees Celsius (i32).
pub struct ThermalHeadroom {
    /// Fraction of thermal budget remaining.
    ///
    /// 0.0 = system is at or above the critical trip point.
    /// 1.0 = system is at ambient temperature (no thermal pressure).
    /// Values between 0.0 and 1.0 are linearly interpolated between
    /// ambient and critical trip point temperatures.
    pub headroom_fraction: f32,

    /// Estimated seconds until throttling begins.
    ///
    /// Based on the current thermal gradient (§9.1). Returns u32::MAX
    /// if no throttling is predicted within the forecast horizon, or if
    /// the system is already in a throttled state.
    pub forecast_seconds: u32,

    /// Current thermal state of the hottest zone.
    pub current_state: ThermalState,

    /// Recommended action for the querying agent.
    pub recommendation: ThermalRecommendation,
}

/// Recommended thermal action for an agent.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ThermalRecommendation {
    /// No action needed. Full workload is sustainable.
    None,

    /// Voluntarily reduce compute intensity.
    ///
    /// The system is approaching a trip point. Reducing workload now
    /// may prevent kernel-imposed throttling later.
    ReduceWorkload,

    /// Defer non-essential background work.
    ///
    /// The system is in a throttled state. Background tasks (prefetch,
    /// cache warming, analytics) should be paused.
    PauseBackground,

    /// Immediately reduce to minimum viable operation.
    ///
    /// The system is at or approaching Critical. The kernel will impose
    /// forced throttling within seconds if the agent does not comply.
    EmergencyReduce,
}
```

**Syscall Interface:**

```rust
// Syscall number: SysThermalHeadroom (future allocation in syscall.rs)
// Signature:
//   x8 = SysThermalHeadroom
//   Returns: ThermalHeadroom written to agent-provided buffer (x0 = pointer)
//   Capability: Capability::ThermalRead required
//   Rate limit: max 10 queries per second per agent
//   Error: IpcError::PermissionDenied if capability absent
//           IpcError::RateLimitExceeded if query rate exceeded
```

The rate limit (10 queries/second) prevents agents from using thermal polling as a high-frequency timer. The Policy Engine maintains a per-agent query counter in the process table and resets it each second.

**Agent Cooperation Model:**

The thermal system distinguishes between cooperative and non-cooperative agents based on their response to thermal headroom recommendations.

- Agents that voluntarily reduce workload when `recommendation != None` receive a scheduling priority boost (scheduling.md §6.2) during the next thermal recovery window
- Agents that ignore `EmergencyReduce` recommendations and continue consuming excess CPU time are subject to forced quota reduction by the Policy Engine, bypassing the normal scheduling class mechanisms
- AIRS tracks per-agent thermal cooperation scores over time; agents with high cooperation scores receive preferential treatment during resource contention (airs.md §5.9)

**Example Agent Usage:**

```rust
// Agent checks thermal headroom before starting compute-intensive work.
// Called at the start of each render loop or batch processing cycle.
fn start_processing_cycle(work: &WorkQueue) {
    let headroom = sys_thermal_headroom()
        .unwrap_or(ThermalHeadroom::default()); // default: None recommendation

    match headroom.recommendation {
        ThermalRecommendation::None => {
            // Full capacity available.
            run_full_pipeline(work);
        }
        ThermalRecommendation::ReduceWorkload => {
            // Reduce quality/complexity but maintain user-visible features.
            run_reduced_pipeline(work);
        }
        ThermalRecommendation::PauseBackground => {
            // Cancel deferred work; run only what the user explicitly requested.
            work.cancel_background_tasks();
            run_essential_only(work);
        }
        ThermalRecommendation::EmergencyReduce => {
            // Minimum viable operation: respond to user input, nothing else.
            work.cancel_all_non_critical();
            run_minimum_viable(work);
        }
    }
}
```

**Headroom Fraction Computation:**

The `headroom_fraction` field is computed from the hottest thermal zone:

```rust
// ambient_mc: measured at last cold boot (i32, millidegrees Celsius)
// critical_mc: zone's critical trip point (i32, millidegrees Celsius)
// current_mc: current zone temperature (i32, millidegrees Celsius)
fn compute_headroom(ambient_mc: i32, critical_mc: i32, current_mc: i32) -> f32 {
    let range = (critical_mc - ambient_mc) as f32;
    let used = (current_mc - ambient_mc) as f32;
    // Clamp to [0.0, 1.0]: 1.0 = at ambient, 0.0 = at or above critical
    (1.0 - (used / range)).clamp(0.0, 1.0)
}
```
