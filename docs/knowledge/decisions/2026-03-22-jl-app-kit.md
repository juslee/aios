# ADR: App Kit for High-Level Application Lifecycle

## Context

BeOS had an Application Kit (BApplication, BMessage, BLooper, BHandler) that managed app lifecycle and messaging. AIOS has IPC Kit and Service Manager for the mechanics, but no high-level "App Kit" that provides app launch, quit, foreground/background transitions, and scripting.

## Options Considered

### Option A: No App Kit — apps use raw IPC Kit + Service Manager

- Pros: Fewer Kits, less abstraction
- Cons: Every app reimplements lifecycle boilerplate, no standard patterns for suspend/resume/quit, no scripting interface, poor developer experience

### Option B: App Kit wrapping IPC Kit + process lifecycle

- Pros: Standard app lifecycle (launch, foreground, background, suspend, resume, quit), scripting interface, consistent patterns across all AIOS apps, BeOS heritage
- Cons: One more Kit to design and maintain

## Decision

App Kit (Option B). Provides high-level application lifecycle management.

Responsibilities:
- App launch, initialization, and shutdown hooks
- Foreground/background transitions
- Suspend/resume (tied to Power Kit)
- Scripting interface (agents and other apps can script actions)
- Standard message dispatch loop (wrapping IPC Kit)

App Kit is an Application Kit — it composes IPC Kit, Capability Kit, and integrates with the Service Manager's process lifecycle.

## Consequences

- Every AIOS native app starts with App Kit as its entry point
- Consistent app behavior across the platform (suspend, resume, quit)
- Scripting enables AIRS and other agents to interact with apps programmatically
- Developer experience improves — less boilerplate, more convention
